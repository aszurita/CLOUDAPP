# DataOps Monitor Runbook

This folder contains the phase 3 Databricks assets for the TPC-DS retail DataOps demo.

## Current Databricks Source

The pipeline expects these Unity Catalog objects to already exist:

- `databricks_proyectobg.tpcds_bronze.store_sales`
- `databricks_proyectobg.tpcds_bronze.date_dim`
- `databricks_proyectobg.tpcds_bronze.item`
- `databricks_proyectobg.tpcds_bronze.store`

Those tables can be generated by the notebook `01_generate_tpcds_data.ipynb` that created the TPC-DS SF1 dataset.

## Job Tasks

Import the notebooks in `/dataops/notebooks` into Databricks and create a Workflow job with this order:

1. `01_ingest_bronze.py`
2. `02_clean_silver.py`
3. `03_publish_gold.py`
4. `04_quality_and_quarantine.py`
5. `05_emit_run_summary.py`

The first task validates the existing Bronze tables. It does not regenerate the TPC-DS dataset.

## Backend Configuration

Configure these values in `CLOUDAPP/backend/.env`:

```env
databricks_host=https://adb-xxxxxxxxxxxxxxxx.azuredatabricks.net
databricks_token=<secret>
databricks_job_id=<job-id>
databricks_catalog=databricks_proyectobg
databricks_schema_bronze=tpcds_bronze
databricks_schema_silver=tpcds_silver
databricks_schema_gold=tpcds_gold
```

The backend never sends the Databricks token to the frontend.

## Portal Flow

1. User clicks `Run Pipeline`.
2. FastAPI calls Databricks Jobs API.
3. Databricks transforms TPC-DS Bronze data into Silver and Gold Delta tables.
4. FastAPI stores run metadata in PostgreSQL.
5. The portal shows Bronze/Silver/Gold counts, quality score, quarantine rows, generated assets and run history.

The current backend starts the Databricks job and records a `running` run with the Databricks link. The next production enhancement is polling the Databricks run output from `05_emit_run_summary.py` and persisting the final real metrics automatically.
